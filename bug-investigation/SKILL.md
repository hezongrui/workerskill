# Bug 排查

## 一句话

配合用户逐步定位 bug 根因。每轮只验证一个假设，把操作、结果、结论、下一步记成可追溯的记录。

## 适用场景

1. 上游 `bug-plan` 已完成，需要进入根因定位。
2. 已经有一两个可能方向，需要通过看代码、加日志、控制变量来验证。
3. 排查要跨多轮对话，每轮都能接上之前状态。

## 不做什么

1. 不做上游事实整理。现象、复现没搞清楚时，先回 `bug-plan`。
2. 不写修复代码、不改生产环境、不改业务配置。
3. 不输出最终修复方案或回归方案。
4. 不在证据不足时强下结论。

## 输入来源

优先读取：

`docs/bugs/[bug-name].md`

没有时先快速补齐必要信息，必要时建议先执行 `bug-plan`。

## 输出路径

`docs/bugs/[bug-name]-investigation.md`

命名示例：

- `docs/bugs/login-token-expire-investigation.md`
- `docs/bugs/api-timeout-under-load-investigation.md`

也可先在对话中推进一轮，后续再汇总写入文件。

## 执行方式

### 每轮只做五件事

1. **看状态**：读上游文档或上一轮记录，确认当前已知事实、可疑模块和假设状态。
2. **审查代码**：根据上游描述扫描可能涉及的模块、函数、调用链、配置项。把怀疑范围写出来，不写根因。
3. **给出排查方向优先级**：基于代码审查结果，列出 1~3 个最可疑方向，按优先级排序（最可能 / 最容易验证 / 影响最大）。
4. **建议最小验证**：从优先级最高的方向开始，从下方验证方法中选一种，给出具体可执行的一步操作。
5. **等反馈更新**：用户执行后返回结果，更新可疑模块、排查方向、假设表和验证记录，再进入下一轮。

### 常用验证方法

| 方法 | 适用场景 | 示例 |
| --- | --- | --- |
| 加日志 | 需要确认运行时状态 | 在 YY 入口打印输入参数和返回值 |
| 控制变量 | 怀疑某个条件触发问题 | 固定 A 不变，只改 B，观察结果 |
| 改配置 | 怀疑配置项影响行为 | 临时调整超时时间、开关、阈值 |
| 复现测试 | 需要稳定触发或缩小范围 | 用最小数据/最小流程复现 |
| 对比版本 | 怀疑近期变更引入 | 回退到上一版本对比是否复现 |

### 控制变量原则

- 一次只动一个条件。
- 明确记录：改了什么、从什么变成什么、观察到什么。
- 如果无法完全隔离变量，要写出混杂因素和限制。

## 每轮开始时的追问

如果接不上状态，先问：

- 上一轮建议的操作做了吗？结果是什么？
- 有没有新的日志、报错、截图或监控数据？
- 现象有没有变化？范围缩小了吗？
- 最近有什么变更？

## 输出要求

1. 每轮只验证一个假设，不要同时抛多个动作让用户自己选。
2. 必须先审查代码：根据上游事实扫描可疑模块、函数、调用链、配置项，不写根因，只写怀疑范围。
3. 审查代码后给出排查方向优先级：按可能性、易验证性、影响面排序，每轮从最高优先级未验证方向开始。
4. 事实和推断分开：观察结果只写看到的，结论写推断。
4. 每轮记录必须包含：目标假设、验证方法、具体操作、观察结果、结论、下一轮建议。
5. 证据不足时明确说"还不能确定"，并给出下一步获取证据的具体建议。
6. 不输出修复代码，但可以给出修复方向（如"建议检查 XX 模块的 YY 逻辑"）。
7. 多轮排查时直接在文件末尾追加新轮次，不要删除旧记录。

## 输出检查清单

- [ ] 上游事实已对齐，缺失项已标注
- [ ] 代码审查结果已记录（可疑模块、函数、调用链、配置项）
- [ ] 排查方向优先级已列出，每轮从最高优先级未验证方向开始
- [ ] 假设表状态随每轮更新
- [ ] 每轮记录包含：目标假设、验证方法、具体操作、观察结果、结论、下一轮建议
- [ ] 结论基于观察结果，没有跳跃推断
- [ ] 下一轮建议具体可执行
- [ ] 没有输出修复代码或修改生产环境的指令
- [ ] 全文白话描述，非技术人员能跟上思路
