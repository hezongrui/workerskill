# PR 交付收口

## 一句话

把“代码写完”收口成“可以稳定提交”的交付包，避免最后一步靠临场发挥。

## 输出路径

默认输出到：

`docs/delivery/YYYY-MM-DD-<feature-name>-delivery.md`

如果用户只要当前对话内输出，可以不落盘。

## 适用边界

适合：
1. 功能开发完成，准备提交 commit 或 PR。
2. bug 修复完成，准备整理测试证据和风险说明。
3. code review 修改后，准备再次提交。

不适合：
1. 需求还没清楚时替代 `plan-prd`。
2. 架构还没定时替代 `design-architect-system`。
3. 代码还没实现时替代 `plan-dev` 或 `test-driven-development`。

## 输入

开始前尽量收集：
1. 当前 diff / PR diff。
2. 对应 PRD、架构文档、开发计划。
3. 相关测试命令和执行结果。
4. 相关 code review 结论和修复情况。

## 执行顺序

1. 确认变更范围  
   只写这次实际改了什么，哪些没改。
2. 自审 diff  
   按正确性、回归、缺测试、资源释放、并发、安全、性能热路径顺序过一遍。
3. 整理测试证据  
   列已运行命令、通过情况、未运行项、无法自动化验证项。
4. 写风险和未覆盖点  
   明确哪些风险仍在，哪些路径本轮没测。
5. 生成提交材料  
   包括 commit message、PR 标题、PR 描述。

## 自审优先级

1. **P0 正确性和回归**：是否可能引入错误结果、崩溃、死锁、资源泄漏、行为回退。
2. **P0 缺失必要测试**：bug 修复、行为变化、边界条件是否有保护。
3. **P1 C++ 生命周期和资源**：所有权、悬垂引用、RAII、异常路径、资源释放。
4. **P1 并发和时序**：竞态、锁顺序、取消、超时、线程退出。
5. **P1 安全和外部输入**：未校验输入、路径/命令拼接、敏感信息泄漏。
6. **P1 性能热路径**：额外拷贝、重复序列化、锁内慢操作、内存搬运。
7. **P2 可维护性**：命名误导、重复逻辑、边界污染。

## Commit / PR 写法

1. 标题先写行为变化，不写过程。
2. 不写“优化了一下”“改了点问题”这种空话。
3. 测试部分只写真实跑过的内容。
4. 风险部分要写剩余风险，不要假装没有。
5. 如果有未覆盖项，要明确说明为什么本轮没测。

## 完成标准

- [ ] 变更范围和未涉及范围清楚
- [ ] 已整理真实测试证据
- [ ] 已列剩余风险和未覆盖点
- [ ] 已完成自审
- [ ] 已生成可直接使用的 commit message
- [ ] 已生成可直接使用的 PR 描述
