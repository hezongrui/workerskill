# onboard-modules

## 说明

对应《老项目快速上手 Playbook》阶段 2：模块拆解 + 优先级总览。

目标是让小白**带着业务场景去看代码**——先知道"这个模块在真实业务里是做什么的"，再顺藤摸瓜找到代码在哪、数据怎么流的、涉及几个线程。

**严禁涉及具体代码实现细节**（算法逻辑、参数列表、条件分支等）。
**允许保留核心类名 + 一句话职责**，以及业务场景中的关键数据格式/协议名称。

## 执行流程（共 3 轮）

### Step 1: 拆分模块清单（第 1 轮）

基于代码库和 `docs/onboarding/architecture.md`（如存在），输出模块清单：

1. 列出所有可独立分析的功能模块（粒度：一个目录或一个服务），总数建议 5~10 个。
2. 每个模块给出：模块名 | 对应目录路径 | 核心文件（2-5个） | 一句话业务职责 | 候选 P0（是/否）。
3. 高耦合模块标注"高耦合，边界模糊"。

### Step 2: 并行分析 Agent（第 2 轮）

对每个模块单元，**并行启动独立子 Agent**。

**重要：Agent 失败处理机制**
- 并行启动后，**逐个检查每个 Agent 的返回结果**，不要一次性等全部返回后才处理。
- 若某个 Agent 返回失败/被拒绝/超时无结果，**主窗口立即重新启动该 Agent**（复用相同 prompt，无需等待其他 Agent）。
- 若同一 Agent 连续失败 2 次，**主窗口降级为直接读取该模块源码**，自己完成分析并写入文档，不再依赖子 Agent。
- 始终保持主窗口上下文：每处理完一个模块（无论通过 Agent 还是主窗口自己），立即更新进度并继续下一个，禁止空等待。

Prompt 模板：
```
你是功能模块分析专家。请仅分析以下单元，不推断其他目录。

分析原则：**先讲业务场景，再讲技术映射**。让小白能回答"这个功能在现实中做什么用的"。

单元名：[单元名]
单元路径：[单元路径]

严格按以下 5 项输出（最终写入 `<模块名>-module-analysis.md`，格式见本文档"最终文档结构"）：

1. 业务场景（必须先说人话）：
   - 这个模块在真实业务中解决什么问题？举 1~2 个具体例子。
   - 必须明确回答：谁在什么场景下会触发这个模块？这个模块最终产出什么给下游？

2. 线程与执行模型（精简，只说数量和职责）：
   - 表格：线程名 | 职责 | 与其他线程如何协作
   - 单线程模块直接写"单线程，在主线程内顺序执行"
   - 不确定标注"待确认"

3. 模块边界与数据流（必须配 Mermaid 流程图）：
   - 节点用业务术语命名，标注触发方、接收方、数据格式/协议
   - 跨线程/跨进程边界用虚线框标注，关键步骤旁标注对应类名
   - **Mermaid 语法严格规范**：
     - 所有 `subgraph` 标题、节点文本、边标签若包含中文/空格/特殊符号，**必须**用双引号包裹（例：`subgraph "TCP接收线程"`、`B["LogRecverTcpComm 监听端口"]`）
     - **禁止**在节点文本或 sequenceDiagram 的 `as` 别名中使用 `<br/>`；换行请拆分为独立节点或直接用空格连接
     - **禁止**使用 HTML 实体（如 `&lt;`/`&gt;`），直接使用原始字符并在必要时用引号包裹
     - sequenceDiagram 的 `participant` 别名只允许纯文本或双引号包裹的字符串，不允许内嵌 HTML 标签

4. 核心文件与入口：
   - 2~5 个关键文件（路径 + 一句话作用）
   - 2~3 个核心类/函数入口（类名+方法名+文件路径+一句话说明）
   - 只给名称和说明，不展开代码

5. 风险与学习优先级：
   - 风险等级（高/中/低）+ 一句话理由
   - 修改频率（高频/中频/低频）
   - 新人建议：先看 / 后看 / 排查问题时再看

约束：只分析 [单元路径]；不猜测外部目录；不确定项标注"待确认"；禁止输出代码片段。
```

### Step 3: 汇总裁决 Agent（第 3 轮）

整合各模块结果，校准模块间依赖、数据流闭环、风险标准一致性。

审查要点：
- 事实准确性：类名、文件路径、线程描述是否正确？
- 业务场景完整性：是否回答了"谁在什么场景下触发、产出什么给下游"？
- 可执行性：新人能否拿着文档直接找到入口文件？

输出：《模块优先级总览》+ 事实性错误清单 + 结论【通过 / 补充修正】

保存文件：
- `docs/onboarding/module-priority-overview.md`
- `docs/onboarding/modules/<模块名>-module-analysis.md`

## 最终文档结构

### `docs/onboarding/module-priority-overview.md`

1. **模块文档导航（置顶）**：
   - 一句话说明："本项目共 N 个模块，每个模块的详细分析见下方链接："
   - 列表：`- [模块名](modules/<模块名>-module-analysis.md) — 一句话业务职责`
   - 让新人一眼知道有几个模块、点哪看详情

2. **项目业务全景回顾**：1~2 句话回顾项目是做什么的（从 architecture.md 摘抄）

3. **端到端业务场景图**：
   - **Mermaid 图可视化**，2~3 个最典型的跨模块业务场景（时序图或流程图）
   - 节点用业务术语，标注模块边界和触发方
   - **Mermaid 语法严格规范**：
     - 所有 `subgraph` 标题、节点文本、边标签若包含中文/空格/特殊符号，**必须**用双引号包裹
     - **禁止**在节点文本或 sequenceDiagram 的 `as` 别名中使用 `<br/>`；禁止 HTML 实体（`&lt;`/`&gt;`）
     - sequenceDiagram 的 `participant` 别名只允许纯文本或双引号包裹的字符串

4. **模块优先级总览**：
   - 表格：模块名 | 业务场景一句话 | 线程数 | 风险 | 修改频率 | 优先级 | 子文档路径 | 新人建议
   - P0 最多 3 个，附理由；P1 最多 3 个；P2 其余

5. **学习路径建议**：
   - 2~3 步学习顺序，每步标注先看哪个模块、从哪个文件入口开始
   - 不要时间线，不要建议时长

### `docs/onboarding/modules/<模块名>-module-analysis.md`

每个模块独立分析，对应 Step 2 的 5 项输出：

1. **业务场景**：1~2 个具体例子
2. **线程与执行模型**：表格（线程名 | 职责 | 协作方式），单线程直接标注
3. **数据流程可视化图**：一张 Mermaid 流程图（业务术语节点、虚线框标跨线程边界、关键步骤旁注类名）
   - **Mermaid 语法严格规范**：
     - 所有 `subgraph` 标题、节点文本、边标签若包含中文/空格/特殊符号，**必须**用双引号包裹
     - **禁止**在节点文本或 sequenceDiagram 的 `as` 别名中使用 `<br/>`；禁止 HTML 实体（`&lt;`/`&gt;`）
     - sequenceDiagram 的 `participant` 别名只允许纯文本或双引号包裹的字符串
4. **模块边界**：承接谁、输出给谁，一句话链路位置
5. **核心文件与入口**：2~5 个关键文件 + 2~3 个核心类/函数入口（均只给路径和一句话说明）
6. **风险与学习优先级**：风险等级 + 理由、修改频率、新人建议
