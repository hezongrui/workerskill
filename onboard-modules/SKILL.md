# onboard-modules

## 一句话

在已经看过项目全局后，把项目拆成模块地图，回答三个问题：每个模块干什么、上下游是谁、先学哪几个最值。

## 适用场景

1. 已经有项目整体印象，准备继续拆模块。
2. 项目模块较多，想先排学习和排查优先级。
3. 后续准备挑一个重点模块继续深挖。

## 不做什么

1. 不展开模块内部代码实现细节。
2. 不替代单模块深挖。
3. 不把所有模块都写成大而全说明书。

## 执行方式

默认使用单 agent 顺序分析。

如果项目模块很多、目录边界清楚，可以按模块分批，或按模块做并行粗扫；但并行只用于收集模块级信息，最终优先级和学习路径仍由主 agent 统一判断。

## 看什么

1. 项目可以拆成哪些独立模块。
2. 每个模块的职责、上下游和链路位置。
3. 哪些模块风险高、改动频率高、值得优先学习。
4. 每个模块推荐从哪些文件或入口开始看。

## 输出路径

保存到：

- `docs/onboarding/module-priority-overview.md`
- `docs/onboarding/modules/<模块名>-module-analysis.md`

## 输出要求

1. 先写业务职责，再写技术映射。
2. 每个模块只写帮助理解和定位的信息。
3. 不确定项明确标注“待确认”。
4. P0 模块最多 3 个，真的重要再进 P0。
5. 总览文档看完后，新人应该知道先学哪几个模块；单模块文档看完后，新人应该知道从哪里入手读代码。
