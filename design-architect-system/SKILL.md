# 通用架构设计

## 一句话

根据需求和业务场景做最小充分的架构设计。先理解问题，再设计边界；先复用现有结构，再考虑新增抽象；只为真实约束和风险增加复杂度。

## 核心原则

1. **业务场景优先**：先说明要解决什么问题、谁使用、主流程是什么，再谈模块和技术。
2. **新老项目分流**：新项目从核心场景推导最小架构；老项目先读现有代码和约定，再做增量设计。
3. **最小充分设计**：能用简单模块解决，不引入层级、框架、服务、队列、缓存或插件系统。
4. **决策可追溯**：关键选择必须写清背景、选择、放弃方案和后果。
5. **只写架构，不写实现代码**：可以写接口契约、数据流、目录结构和伪流程，不写函数体、SQL 或可运行代码。
6. **面向落地**：输出必须能指导 `plan-dev` 继续拆开发顺序。

## 入口判断

先判断项目类型，再读取对应参考：

1. **新项目 / 从零开始**：读取 `references/new-project.md`。
2. **老项目 / 现有代码增量**：读取 `references/existing-project.md`。

如果不确定，优先按老项目处理：先扫描当前代码库，再决定是否需要新项目设计。

## 架构输入

开始设计前尽量收集：

1. 需求文档或用户口述目标。
2. 核心业务场景和主流程。
3. 明确的非功能约束：性能、并发、可靠性、安全、兼容性、部署、团队规模。
4. 现有代码结构、构建系统、测试方式和已采用技术栈。
5. 用户明确不想做或当前阶段不需要做的内容。

信息不足时先列假设，不要用复杂架构填补未知。

## 防过度设计规则

默认不要引入下面设计，除非需求或现有代码给出明确理由：

1. 没有多团队独立交付和独立扩缩容需求，不拆微服务。
2. 没有异步削峰、可靠投递、跨系统解耦需求，不引入消息队列。
3. 没有性能瓶颈、跨请求复用或明确容量压力，不加缓存。
4. 没有复杂业务不变量和长期演进的领域模型，不硬套 DDD。
5. 没有稳定扩展点和第三方扩展需求，不设计插件系统。
6. 没有 profile、容量指标或明确热路径，不为了“可能更快”复杂化。
7. 没有跨进程或跨团队边界，不急着抽 RPC/API 层。
8. 没有明确审计、排障或运维需求，不把日志系统作为架构前置条件。

如果确实要引入复杂组件，必须写清触发条件和不引入的后果。

## 技术选型规则

1. 老项目已有技术栈：默认沿用，除非它已经阻碍需求落地。
2. 新项目技术明确：直接使用，只补充风险和替代方案。
3. 技术不确定、引入新依赖、影响长期维护或安全合规时，才联网调研。
4. 调研只服务决策，不堆资料。输出 1 个推荐方案、1 到 2 个放弃方案和理由。

## 输出文档

默认保存到：

`docs/architect/architect-[project-or-feature].md`

老项目增量可以使用：

`docs/architect/architect-[project]-increment-[feature].md`

如果项目已有架构文档命名规范，优先遵循项目约定。

## 输出结构

架构文档必须包含：

1. **目标和范围**：解决什么问题，不解决什么问题。
2. **业务场景**：2 到 5 个关键场景，按优先级排序。
3. **架构驱动因素**：真正影响架构的约束和质量属性。
4. **现状判断**：新项目说明从零约束；老项目说明现有结构和复用点。
5. **模块设计**：模块职责、依赖方向、输入输出、边界契约。
6. **数据流 / 调用流**：核心路径如何经过模块。
7. **关键决策**：用 ADR-lite 表格记录。
8. **目录结构**：只列和本次架构相关的目录或文件。
9. **风险和回退**：触发条件、影响、应对、回退。
10. **测试切入点**：哪些模块适合单元、集成、手动或专项测试。
11. **交给 `plan-dev` 的开发边界**：新增、修改、复用的模块清单。

## C++ 项目额外关注

如果是 C++ 项目，架构设计必须额外说明：

1. 模块边界和头文件依赖，避免公共头膨胀和循环 include。
2. 对象所有权和生命周期，尤其是指针、引用、`string_view`、`span`、回调。
3. 线程模型：谁创建线程，谁停止线程，跨线程数据如何同步。
4. 错误传播：异常、错误码、`optional/expected` 或项目既有风格。
5. ABI/API 兼容：公开头文件、虚函数、结构体布局、枚举和序列化格式。
6. 性能路径：大对象拷贝、内存搬运、序列化、锁粒度、IO 边界。
7. 测试边界：哪些逻辑能单元测试，哪些需要 fake、临时文件、集成环境或手动测试。

## 完成标准

输出前检查：

1. 架构是否从业务场景和约束推导出来。
2. 是否区分了新项目和老项目增量。
3. 是否明确哪些不做。
4. 是否避免了无依据的复杂组件。
5. 是否能直接交给 `plan-dev` 拆开发顺序。
6. 是否没有写实现代码。
