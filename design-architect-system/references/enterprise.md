# Enterprise 模式

目标：6个月+周期，5+人团队，多团队协作。
原则：领域驱动，清晰边界，完整可观测性。

## 架构特征

- 模块数：10+ 个/服务
- 复杂度：高，多团队协作
- 可靠性：容错设计，性能预算，可观测性
- 测试：单元 + 集成 + 契约 + E2E

## 4. 目录结构

```
services/
├── gateway/                # API 网关：认证、路由、限流
├── service-a/              # 领域 A 服务
│   ├── src/
│   │   ├── application/    # 应用层：用例编排
│   │   ├── domain/         # 领域层：实体、值对象、领域事件
│   │   └── infra/          # 基础设施：仓库实现、消息发布
│   └── tests/
├── service-b/              # 领域 B 服务（结构同 service-a）
├── shared/                 # 共享库：契约、工具、中间件
├── deployment/             # K8s / Terraform / CI 配置
└── observability/          # 监控、日志、追踪配置
```

| 目录 | 功能说明 |
|------|----------|
| `gateway/` | 统一入口，负责鉴权、路由、流量控制 |
| `service-*/src/application/` | 用例编排、事务控制、DTO 转换 |
| `service-*/src/domain/` | 核心业务规则、领域模型、领域事件 |
| `service-*/src/infra/` | 数据库访问、外部服务调用、消息发布 |
| `shared/` | 跨服务复用的契约、工具、通用中间件 |
| `deployment/` | 容器编排、基础设施即代码、流水线脚本 |
| `observability/` | 监控大盘、日志聚合、分布式追踪配置 |

## 设计原则

1. 领域优先，再定服务边界
2. 康威定律：架构匹配组织
3. 数据主权：服务拥有各自数据
4. 显式接口：契约明确，版本管理
5. 可观测性：日志、指标、追踪三位一体
6. 容错：熔断、降级、限流、重试
7. 从模块化单体开始，演进式拆分
