# Increment 模式（老项目加功能）

原则：融入现有架构，不另起炉灶，明确新增/修改/复用。

## 架构特征

- 模块数：1-5 个新增/修改模块
- 核心目标：让新功能平滑融入现有代码库
- 约束：必须遵守现有项目的目录规范、依赖方向和命名风格

## 输出模板

```markdown
# 增量架构设计: [项目名] - [新增功能] (Increment)

## 1. 现有架构速览

- **项目类型**: [Quick / Standard / Enterprise]
- **技术栈**: [语言/框架/数据库]
- **现有核心模块**:
  - `模块A`：...
  - `模块B`：...

### 现有目录结构（精简版）

```
project/
├── app/                    # 现有：业务代码
├── core/                   # 现有：配置/日志
└── tests/                  # 现有：测试
```

## 2. 增量需求

- **目标**：一句话说明本次要新增的功能
- **涉及范围**：新增模块 + 需要改动的现有模块

## 3. 增量模块设计

### 模块关系

```
[现有模块A] → [新增模块X] → [现有模块B]
                ↓
           [新增模块Y]
```

### 模块清单

| 模块 | 标记 | 职责 | 依赖 |
|------|------|------|------|
| `新增模块X` | 【新增】 | ... | 现有模块A |
| `现有模块B` | 【修改】 | 增加接口以接收新增模块X的输出 | 新增模块X |
| `现有模块A` | 【复用】 | 直接复用，无需改动 | - |

## 4. 目录结构（含增量标注）

```
project/
├── app/
│   ├── existing.py         # 【复用】现有模块
│   ├── new_feature.py      # 【新增】新增功能主模块
│   └── handlers/
│       └── __init__.py     # 【修改】注册新路由/处理器
├── core/
│   └── config.py           # 【修改】增加新配置项
└── tests/
    └── test_new_feature.py # 【新增】新增测试
```

| 目录/文件 | 标记 | 功能说明 |
|-----------|------|----------|
| `app/new_feature.py` | 【新增】 | 新增功能的核心实现 |
| `app/handlers/__init__.py` | 【修改】 | 注册新路由或事件处理器 |
| `core/config.py` | 【修改】 | 增加与新功能相关的配置 |
| `tests/test_new_feature.py` | 【新增】 | 覆盖新增功能的测试用例 |

## 5. 集成点与改造点

| 集成点 | 现有状态 | 改造方式 | 涉及文件 |
|--------|----------|----------|----------|
| 路由注册 | 手动注册 | 在 `handlers/__init__.py` 中增加新路由 | `app/handlers/__init__.py` |
| 配置读取 | 从 `.env` 读取 | 在 `core/config.py` 中增加新字段 | `core/config.py` |

## 6. 风险

| 风险 | 触发条件 | 影响 | 应对 | 回退 |
|------|----------|------|------|------|
| 破坏现有功能 | 修改了公共接口 | 现有测试失败 | 修改前跑通全部测试；小步修改 | 回滚变更 |
| 目录风格不统一 | 新增模块命名与现有规范冲突 | 代码混乱 | 严格遵循现有命名约定 | 重命名 |
```

## 设计原则

1. **先读懂现有代码**，再设计增量
2. **新增模块必须放在现有目录体系内**，不要开新根目录
3. **修改现有文件要最小化**，优先通过新增文件扩展
4. **明确标注每个文件的标记**：【新增】/【修改】/【复用】
5. **增量架构必须跑通现有测试 + 新增测试**
