# Standard 模式

原则：清晰分层，接口预留，不过度设计。

## 架构特征

- 模块数：4-8 个
- 分层：明确，便于协作
- 扩展性：接口预留，实现先简单
- 测试：核心路径单元测试 + 关键集成测试

## 4. 目录结构

```
project/
├── app/                    # 应用层：路由、控制器、依赖注入
│   ├── api/                # API 接口定义
│   ├── services/           # 业务逻辑
│   └── models/             # 领域模型 / ORM
├── core/                   # 核心层：配置、日志、异常、中间件
├── infra/                  # 基础设施：数据库、缓存、消息队列客户端
├── tests/                  # 测试（按单元/集成分层）
└── docs/                   # 架构/接口文档
```

| 目录 | 功能说明 |
|------|----------|
| `app/api/` | 接收请求、参数校验、返回响应 |
| `app/services/` | 核心业务逻辑编排 |
| `app/models/` | 数据模型、领域对象定义 |
| `core/` | 跨层共享的基础能力（配置、日志、异常处理） |
| `infra/` | 外部资源接入（DB、缓存、MQ、第三方 SDK） |
| `tests/` | 单元测试 + 集成测试 |
| `docs/` | 设计文档、API 说明 |

## 设计原则

1. 分层清晰，上层不依赖下层实现
2. 依赖注入，便于测试
3. 接口预留扩展，实现先简单
4. 核心业务必须有单元测试
5. 三次以上重复再提取抽象
