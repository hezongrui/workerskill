# onboard-focus

## 说明

本 skill 对应《老项目快速上手 Playbook》阶段 3：单模块重点深挖。

目标：从面面俱到的模块文档中，提炼出 **2~5 个真正需要啃透的核心功能点**，并给出**可验证的日志点**，支撑阶段 4 的最小实战修改。

**核心原则**：每个重点必须能回答——「**什么业务场景会触发这条代码路径？**」

## 前置要求

执行本 skill 前，必须明确指定要深挖的模块名，且对应模块分析文档必须已存在。

- **如果用户已指定模块名**：检查 `docs/onboarding/modules/<模块名>-module-analysis.md` 是否存在。
  - 若存在：直接进入执行流程。
  - 若不存在：停止分析，提示用户：「该模块尚未生成模块分析文档，请先执行 onboard-modules Skill。」
- **如果用户没有指定具体模块**：使用 `AskUserQuestion` 工具向用户确认：
  > "请指定要深挖的模块名称（该模块需已存在于 `docs/onboarding/modules/` 目录中）。"

## 执行流程（共 3 轮）

### Step 1: 主 Agent - 模块重点提炼

请以"模块重点提炼专家"角色，结合以下输入：
1. 用户指定的模块名及对应路径
2. 已存在的 `docs/onboarding/modules/<模块名>-module-analysis.md`

对该模块的实际源码进行逐行核对，输出《模块重点掌握清单》初稿。

**对 2~5 个核心功能点中的每一个**，必须给出：

| 字段 | 说明 |
|------|------|
| **重点名称** | 用业务语言命名，如「日志接收后的分包重组」，而非 `ParseLogPacket()` |
| **业务触发场景** | **必填**。什么人在什么操作下会走到这里？例如：设备批量上报日志时，TCP 粘包导致多个日志包合并到达，此逻辑负责拆分。 |
| **涉及类名 + 方法名** | 必须准确，给出文件路径 |
| **核心代码块** | 只贴最关键的 **5~15 行**代码。若涉及 C++ 构造/析构、锁范围、RAII 等资源管理惯用法，可放宽到 15 行，但必须在代码块上方用 1 句话说明「这段代码的核心看点是什么」。禁止大段粘贴。 |
| **为什么这是重点** | 在整体业务链路中的作用；如果失败/异常，对下游有什么影响 |
| **建议看代码顺序** | 第一步看 X 文件，第二步看 Y 文件的 Z 函数 |
| **辅助性标注** | 如只是辅助性的，标注【了解即可】 |

**可视化要求**：
- **模块在端到端链路中的位置**：若该模块涉及 ≥2 个外部交互方或跨线程/跨进程边界，则必须配一张极简 Mermaid 流程图/序列图，标注输入来源、输出去向、边界上的核心类名；若模块极简（单类单循环、无外部交互），可用 2~3 句话文字描述替代。
- **可选**：若某个核心功能点涉及复杂状态流转或多组件协作，可另配一张极简时序图（节点用业务术语，禁止罗列函数调用）。

**可验证日志点**：针对该模块，给出 **1~2 个** 适合阶段 4 做最小日志修改的具体位置。每个点必须包含：
- 文件路径 + 具体类名/函数名 + 建议插入的大致行号（或已有日志行的行号）
- 建议打印的内容（如入口参数、状态值）
- **预期输出示例**：必须参考该文件中**已有日志的风格**（前缀格式、时间戳样式、字段命名习惯），不要编造项目中不存在的日志格式
- 为什么选这个位置（能验证哪条数据流或分支）

> 约束：必须是安全的日志修改点，不涉及业务逻辑变更。

**禁止**：
- 不要输出与 module-analysis.md 重复的内容（如完整的线程模型说明、目录结构）。
- 不要面面俱到，只输出"浓缩后的重点"。

### Step 2: 子 Agent - 定位与逻辑审查

请以"严格的模块评审专家"角色，对《模块重点掌握清单》初稿进行审查，**只挑硬伤，不修改文档**：

1. **定位准确性**：类名、方法名、文件路径、建议行号是否与源码一致？
2. **逻辑一致性**：代码块是否确实支撑所述的数据流或分支逻辑？业务触发场景描述是否与代码路径匹配？
3. **可执行性**：新人能否按照"建议看代码顺序"直接找到文件并理解？可验证日志点是否安全、是否真的能验证所述数据流？预期输出示例是否贴合已有日志风格？
4. **可视化有效性**：Mermaid 图是否准确反映模块边界？节点命名是否混淆了业务术语与技术术语？

输出：
- 事实性错误清单（引用原文 + 正确信息 + 代码位置证据）
- 待确认项清单
- 结论：【通过 / 需修正】

### Step 3: 裁决 Agent - 输出最终清单

请以"资深技术负责人/裁决 Agent"角色，综合初稿和 Step 2 的审查意见，输出最终版《模块重点掌握清单》。

- 若 Step 2 结论为【通过】：直接润色并输出终稿。
- 若 Step 2 结论为【需修正】：逐项修正硬伤，删除存疑内容，必要时标注「待人工复核」。

最终文档要求：
- 只保留确定无误的事实和可执行的建议
- 删除所有推测性、不确定性的描述
- 确保每个核心功能点的代码位置足够精确

保存文件：`docs/onboarding/modules/<模块名>-key-points.md`

## 人类下一步行动指引

> 下一步是**最小可验证修改 + 调试（人执行）**：
> 1. 打开本清单的「可验证日志点」，按表修改一处日志。
> 2. 编译/运行，观察日志输出是否与「预期输出示例」一致。
> 3. 在核心功能点的入口函数打断点，命中后观察调用栈和局部变量。
> 4. **发现文档与代码不符，立即修正。**
>
> 目的：用真实代码修改和日志观察"校准"文档准确性。
