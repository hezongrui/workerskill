# CI 失败分诊

## 一句话

先找到第一有效失败点，再判断它属于哪一类问题，不被后续连锁报错带偏。

## 输出路径

默认输出到：

`docs/triage/YYYY-MM-DD-<issue-name>-ci-failure.md`

## 适用边界

适合：
1. CMake / Ninja / MSBuild / Make 构建失败。
2. 链接失败、符号冲突、ABI 不兼容。
3. 单元测试、集成测试、端到端测试失败。
4. 环境、依赖、权限、路径、工具链问题。
5. 偶发失败和 flaky test。

不适合：
1. 替代 `code-review` 做代码质量审查。
2. 替代 `error-retrospective` 做长期复盘。

## 核心规则

1. 先找第一有效失败点，不先看最后一行。
2. 区分“根因报错”和“连锁报错”。
3. 先分类，再给修复建议。
4. 没证据时写“待确认”，不要假装定位完成。

## 分类顺序

1. **P0 编译失败**：语法、头文件、模板实例化、类型不匹配、宏条件、平台差异。
2. **P0 链接失败**：未定义符号、重复定义、库顺序、ABI 不兼容、导出缺失。
3. **P0 测试失败**：断言失败、行为回归、边界条件、竞态、超时。
4. **P1 环境问题**：路径、权限、依赖缺失、工具版本、证书、网络、磁盘空间。
5. **P1 Flaky / 时序问题**：偶发超时、顺序依赖、共享状态污染、随机种子。
6. **P2 流水线配置问题**：脚本错误、参数错误、缓存污染、工件缺失。

## C++ 重点排查

1. 编译报错从第一个报错文件和第一个模板展开点看起。
2. 链接报错先看符号归属、声明定义是否一致、库是否进了链接命令。
3. 平台差异关注编译器版本、标准版本、宏分支、Windows/Linux 路径和编码。
4. 测试失败关注未初始化、生命周期、并发、时钟、随机数、资源释放。
5. flaky 先看共享目录、全局状态、时间依赖、线程调度和网络依赖。

## 处理纪律

1. 日志长时，只摘第一有效失败链，不抄整段。
2. 测试挂了先看测试本身，再看环境，不默认是代码错。
3. 编译和链接问题优先给本地最小复现命令。
4. 不能稳定复现时，要明确标注为疑似 flaky。
