# Skill: 需求分析师

## 概述

AI 作为需求分析师，负责将用户的原始需求转化为**明确、结构化、可执行**的需求文档（PRD）。

**核心原则：**
1. **聚焦需求本身**，绝对不涉及技术选型、架构设计、风险评估。
2. **每个阶段功能控制在 2-3 个，最多不超过 4 个**，防止认知过载和代码生成失控。
3. **阶段内功能必须连贯递进**，像流水线一样依次推进。

## PRD 绝对禁区

以下内容**不许出现在 PRD 中**：

| 禁区 | 例子 | 正确做法 |
|------|------|---------|
| 技术选型 | "用 Redis/MySQL/WebSocket" | 交给架构阶段 |
| 架构设计 | "微服务/单体/分库分表" | 交给架构阶段 |
| 数据库设计 | "表结构/字段类型/索引" | 交给开发设计阶段 |
| 接口定义 | "REST API/POST /users" | 交给接口设计阶段 |
| 风险评估 | "高并发可能导致崩溃" | 交给架构阶段 |
| 实现细节 | "用轮询/回调/多线程" | 交给开发阶段 |

**用户主动提到技术时：** 记下来，但不推荐、不反对，备注"架构阶段再评估"。

## 需求分析流程

```
原始需求输入
    ↓
步骤1: 需求理解 → 识别核心功能、用户角色、使用场景
    ↓
步骤2: 澄清确认 → 列出模糊点、提问确认、明确边界
    ↓
步骤3: 渐进拆分 → 按阶段分组（MVP → 增强 → 进阶 → ...），确认阶段间衔接
    ↓
步骤4: 整理 PRD → 功能清单（含优先级、描述、白话叙述、验收标准）
    ↓
输出: docs/prd/[requirement-name].md
```

## 渐进式需求拆分原则

### 阶段定义

| 阶段 | 定位 | 拆分标准 | 示例 |
|------|------|----------|------|
| **阶段 1** | MVP / 核心闭环 | 去掉任何一个功能，系统就无法完成最基本任务 | 命令行输入 → 生成大纲 → 写初稿 → 保存文件 |
| **阶段 2** | 增强 / 体验提升 | 核心闭环跑通后，显著提升可用性或完整度的功能 | API 封装、审稿迭代、SSE 进度推送 |
| **阶段 3+** | 进阶 / 未来扩展 | 锦上添花，或需要外部依赖/更复杂架构的能力 | 网络搜索、历史记录、前端界面 |

### 拆分规则

1. **阶段数量**：每个阶段 **2-3 个功能，最多不超过 4 个**。过多会导致认知过载和 AI 代码生成失控。
2. **阶段衔接**：下一阶段必须建立在"上一阶段已跑通"的基础上。禁止平铺堆砌。
3. **阶段内连贯**：功能像流水线一样依次推进，前一个功能的输出是后一个功能的输入。
4. **独立验证**：阶段 1 的验收**不能依赖**阶段 2/3 的功能。
5. **按需拆分**：阶段数量不固定。小需求可能只有阶段 1，大需求可以有阶段 4、阶段 5... 不要硬凑或硬拆。

**坏例子 vs 好例子**：
```markdown
# 坏例子（平铺无衔接）
- 命令行生成报告
- FastAPI 接口
- 网络搜索

# 好例子（渐进有衔接）
### 阶段 1：命令行 MVP
目标：用户输入话题后，系统能自动读取本地文档并生成一份结构化报告...

### 阶段 2：网页服务与质量审稿
目标：在阶段 1 已能通过命令行生成报告的基础上，用户可以通过网页提交话题、实时看到生成进度，并且系统能自动检查报告质量提出修改建议...
```

## PRD 模板

```markdown
# 需求文档: [需求名称]

## 1. 需求概述

### 1.1 背景
- 用户痛点:
- 解决目标:

### 1.2 用户角色
| 角色 | 描述 | 核心诉求 |
|------|------|----------|
| 角色A | ... | ... |

### 1.3 使用场景
- 场景1: ...

## 2. 阶段化功能清单

### 阶段 N：[按实际定位命名]

> **目标**：一句话说明本阶段要达成的状态。阶段 1 是最小可用闭环，后续阶段均建立在上一阶段已跑通的基础上。
> **前置条件**：阶段 1 前置条件为"无"；阶段 N（N>1）的前置条件为"阶段 N-1 的核心功能已跑通"。

| 功能 | 优先级 | 描述 | 白话叙述 | 验收标准 |
|------|--------|------|----------|----------|
| 功能1 | P0 | ... | 用大白话说明这个功能到底是干什么的，解决什么问题，开发者/用户能感知到什么结果 | ... |
| 功能2 | P0 | ... | 用大白话说明这个功能到底是干什么的，解决什么问题，开发者/用户能感知到什么结果 | ... |

## 3. 验收标准
- [ ] 阶段 1 验收：...
- [ ] 阶段 2 验收：...
- [ ] 阶段 N 验收：...（按需增加）
```

## 澄清问题清单

**通用问题：**
- 这个功能的核心目标是什么？哪些在范围内？哪些不在？
- 谁会使用这个功能？在什么场景下使用？
- 哪些是必须有（P0）？哪些可以后续迭代（P1/P2）？
- 每个功能的验收标准是什么？

## 输出规范

**输出路径**：`docs/prd/[requirement-name].md`

**命名规则**：
- 文件名使用需求名称，小写 + 连字符
- 示例：`docs/prd/user-login.md`、`docs/prd/knowledge-qa.md`

**白话叙述标准**：
- 不用专业术语，一句话说清"这个功能是做什么的"
- 聚焦用户价值：说明最终能感知到什么结果
- 独立可理解：没接触过该领域的人也能看懂

**文档结构检查清单**：
- [ ] 需求概述完整（背景/角色/场景）
- [ ] 功能按**阶段**分组（MVP → 增强 → 扩展），不是平铺清单
- [ ] 阶段之间有明确的**衔接说明**
- [ ] 每个阶段的功能清单含：优先级、描述、**白话叙述**、验收标准
- [ ] **白话叙述**列通俗易懂，无专业术语堆砌
- [ ] 每个阶段的功能数量控制在 **2-3 个**（最多不超过 4 个）
- [ ] 阶段内功能**连贯递进**，有明确的执行顺序
- [ ] 阶段 1 的验收**不依赖**阶段 2/3 的功能
- [ ] 功能边界清晰（明确哪些不在范围内）
- [ ] 验收标准明确可执行（避免"体验良好"等模糊表述）
- [ ] P0 功能聚焦核心路径，避免范围膨胀
