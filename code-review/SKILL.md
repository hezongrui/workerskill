# 代码审查

## 一句话

先找会导致真实故障的风险，再看测试和维护性。审查输出必须 findings-first：有问题先列问题，没问题就明确说没有发现必须修改的问题。

## 审查边界

默认只审查，不改代码。除非用户明确要求修复，否则不要直接编辑文件。

## 保存路径

需要落文档时，保存到：

`docs/reviews/YYYY-MM-DD-<审查对象>-code-review.md`

如果用户只要求当前对话审查，可以直接在对话中输出 findings。

优先审查：
1. 当前 `git diff` / PR diff。
2. diff 触达的调用链和公共接口。
3. 相关测试、构建脚本、配置和文档。
4. C++ 头文件变更带来的 ABI/API、生命周期和依赖影响。

不要把下面内容当成高优先级问题：
1. 单纯个人风格偏好。
2. 项目既有风格已经接受的写法。
3. 没有实际失败路径的理论洁癖。
4. 只靠猜测、没有代码证据的问题。

## 审查优先级

按下面顺序找问题：

1. **P0 正确性 / 回归**：会导致崩溃、数据损坏、错误结果、死锁、资源泄漏、线上行为回退。
2. **P0 安全问题**：越界、UAF、double free、命令/路径注入、敏感信息泄露、权限绕过、不可信输入未校验。
3. **P1 缺测试**：行为变化、bug 修复、边界条件、错误路径没有测试保护。
4. **P1 C++ 生命周期和资源管理**：所有权不清、悬垂引用、`string_view`/指针失效、异常安全、RAII 破坏。
5. **P1 并发和时序**：数据竞争、锁顺序、死锁、原子内存序、超时/取消、线程生命周期。
6. **P1 API / ABI 兼容**：公开头文件、虚函数、结构体布局、枚举语义、错误码、序列化格式改变。
7. **P1/P2 性能和容量**：热路径、大数据、实时链路中的额外拷贝、重复序列化、锁粒度扩大、N+1、内存峰值、无界队列、资源耗尽。
8. **P2 可维护性**：重复复杂逻辑、错误抽象、难以测试、命名误导、模块边界被污染。
9. **P3 风格和一致性**：只在项目已有明确规范时提出。

## C++ 重点清单

审 C++ 时优先看这些点：

1. 生命周期：对象、引用、指针、迭代器、`std::string_view`、`std::span` 是否可能悬垂。
2. 所有权：裸指针是否表达非拥有关系，拥有关系是否用 RAII / 智能指针表达。
3. 拷贝和移动：Rule of Zero/Five、移动后对象状态、自定义析构和拷贝是否成套。
4. 容器失效：`vector` 扩容、erase、并发访问、返回内部 buffer 是否安全。
5. 错误处理：异常、错误码、`expected`/`optional` 风格是否与项目一致，失败路径是否可观察。
6. 异常安全：资源释放、部分写入、回滚语义、析构函数是否抛异常。
7. 并发：共享状态是否同步，锁顺序是否稳定，回调中是否重入，线程退出是否 join/cancel。
8. 整数和边界：溢出、符号转换、size 截断、数组/缓冲区边界。
9. C API 交互：缓冲区长度、所有权转移、错误码检查、编码和空终止。
10. 构建影响：头文件包含膨胀、ODR、宏污染、平台条件编译。

## 性能审查清单

性能问题要结合场景判断，不要把微优化当成必须修改的问题。

优先级判断：
1. **P1 必须关注**：热路径、实时链路、大文件/大消息、高频循环、跨线程/跨进程传输中出现明显额外拷贝或重复搬运。
2. **P1 必须关注**：改动引入性能回退，例如原本移动变拷贝、原本流式处理变整块加载、原本 O(n) 变 O(n²)。
3. **P2 建议优化**：有较大概率收益，但当前 diff 无法证明是瓶颈，需要 benchmark、profile 或压测验证。
4. **P3 暂不处理**：只是不在热路径上的小对象拷贝、可读性会明显变差的微优化、没有数据支撑的“可能更快”。

C++ 性能重点看：
1. 是否可以避免不必要的深拷贝，改用移动、引用、`std::span`、`std::string_view` 或流式处理。
2. 零拷贝是否有清晰生命周期保证；如果生命周期不清，宁可拷贝，不要制造悬垂引用。
3. 跨模块/跨线程/跨进程是否反复序列化、反序列化、打包、拆包。
4. 容器是否反复扩容，是否需要 `reserve`、批量插入或更合适的数据结构。
5. 热路径是否频繁分配内存、构造临时对象、格式化字符串或写同步日志。
6. 锁粒度是否扩大，是否把慢 IO、回调或复杂计算放进锁内。
7. 大对象参数是否按值传递，返回值是否能依赖 NRVO/move，接口是否强迫调用方复制。
8. 性能建议最好给出验证方式：benchmark、profile、计数器、trace 或压测场景。

## 安全审查清单

安全先并入本 skill，不单独拆 skill。只有项目长期涉及认证、支付、文件上传、设备控制或外部暴露 API 时，才考虑独立 `security-review`。

重点看：
1. 不可信输入：文件、网络、CLI、环境变量、设备消息、第三方 SDK 返回值是否校验。
2. 内存安全：越界、UAF、double free、未初始化读取、格式化字符串风险。
3. 注入风险：命令拼接、路径拼接、SQL/脚本/表达式拼接、动态库加载路径。
4. 权限和状态：鉴权、授权、状态机绕过、重放、幂等、TOCTOU。
5. 敏感信息：token、密钥、个人信息是否进入日志、错误消息、dump、测试 fixture。
6. 资源滥用：无界循环、无界缓存、压缩炸弹、大文件、无限重试、缺少超时。
7. 加密随机：不要手写 crypto；随机数用途要区分安全随机和普通随机。

## 测试审查

测试意见必须落到具体风险：

1. bug 修复必须有回归测试。
2. 行为变化必须有能失败再变绿的测试。
3. C++ 生命周期、错误路径、边界条件、并发/超时要有针对性测试或说明为什么不能自动测。
4. 外部依赖优先测自己的封装和协议，不测第三方库本身。
5. 不要求为了覆盖率数字补低价值测试。

## 工具信号

审查可以建议使用工具，但不要把工具输出原样当最终结论：

1. C++ 静态分析：`clang-tidy`、编译器 warning、MSVC Code Analysis、cppcheck。
2. 动态检查：ASan、UBSan、TSan、LSan。
3. 构建和测试：CMake/CTest、项目已有 CI、相关单元测试和集成测试。
4. PR 自动化：reviewdog 适合把 linter/static analyzer 输出限制在 diff 上。

## 输出格式

有问题时，只输出 findings，再给简短总结。按严重程度排序：

```text
1. [P0/P1/P2/P3] 标题
   位置：path/to/file.cpp:123
   问题：说明具体错误，不写泛泛建议。
   触发场景：给出能发生问题的输入、状态或调用顺序。
   建议：给出最小修复方向或需要补的测试。
```

findings 后补一个处理优先级，帮助用户抓重点：

```text
处理优先级：
1. 必须先修：P0，以及会导致明确回归的 P1。
2. 本轮建议修：缺测试、生命周期、并发、接口兼容、热路径性能问题。
3. 可排期：需要 profile/benchmark 验证的性能优化、较低风险的维护性问题。
4. 可忽略：无证据的微优化、纯风格偏好。
```

没有发现必须修改的问题时：

```text
未发现必须修改的问题。

剩余风险：
1. 哪些测试没有运行。
2. 哪些路径无法从当前 diff 判断。
3. 哪些问题需要 CI、sanitizer 或集成环境验证。
```

## 审查纪律

1. 不要先夸代码。
2. 不要把摘要放在 findings 前面。
3. 每个问题必须能指向具体文件和代码。
4. 不确定时写成“需要确认”，不要装成确定缺陷。
5. 同类问题合并，不要刷屏。
6. 如果问题只是缺测试，说明缺哪条行为测试。
7. 如果用户让你修复，先列将要改的点，再进入修改。
