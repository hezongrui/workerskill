# 测试计划设计

## 一句话

把一个模块或功能拆成可执行、可验收、可回归的测试用例表。测试方式可以是自动化，也可以是手动测试，但每条都必须有明确步骤和预期结果。

## 适用范围

适合：
1. 接手一个模块后，需要知道整体怎么测。
2. 功能开发完成后，需要提测或做回归测试。
3. 代码审查发现缺测试，需要补测试设计文档。
4. C++ 项目涉及文件、网络、进程、设备、性能、并发、长稳等不方便全部单元测试的场景。

不适合：
1. 替代 `test-driven-development` 写开发期单元测试。
2. 替代 `code-review` 查具体 diff 缺陷。
3. 为私有函数逐个列测试。
4. 写“看起来正常”这种不可验收的手动测试。

## 保存路径

默认保存到：

`docs/test-plans/[module-or-feature]-test-plan.md`

如果已有项目测试文档目录，优先按项目约定保存。

## 前置输入

开始前先尽量读取：
1. PRD：`docs/prd/`
2. 架构或模块设计：`docs/architect/`
3. 开发计划：`docs/plans/`
4. 相关源码入口、公开 API、CLI、配置文件和已有测试。
5. README、CI 配置、构建脚本中的测试命令。

没有文档时，直接从代码入口和用户可见行为反推功能，不要凭空编造需求。

## 设计流程

1. 明确测试对象：模块、功能、接口、命令或流程。
2. 列出功能点：只列用户可见行为、公开接口行为和边界契约。
3. 按风险排序：P0 主路径和高风险异常先测，P1 边界和回归，P2 兼容和性能，P3 低风险补充。
4. 为每个功能点设计用例：正常、边界、异常、重复调用、取消/超时、资源释放。
5. 标记测试方式：单元测试、集成测试、端到端测试、手动测试、性能测试、长稳测试。
6. 写清观测证据：返回值、退出码、文件、数据库、消息、日志、指标、性能数据、界面状态。
7. 给自动化建议：哪些必须自动化，哪些可以手动，哪些需要后续压测或专项验证。

## 优先级定义

1. **P0 必测**：核心主路径、线上 bug 回归、数据正确性、安全边界、崩溃/资源泄漏风险。
2. **P1 应测**：关键异常、边界值、兼容性、并发/超时、接口契约。
3. **P2 建议测**：性能容量、长稳、可观测性、非核心组合路径。
4. **P3 可选**：低风险显示文案、重复覆盖、探索性测试。

## C++ 测试关注点

对 C++ 模块额外关注：
1. 生命周期和资源释放：文件句柄、线程、socket、设备连接、临时文件、内存峰值。
2. 错误路径：错误码、异常、`optional/expected`、退出码是否符合契约。
3. 边界输入：空输入、超长输入、非法格式、大文件、截断数据、重复字段。
4. 并发时序：重复启动、取消、超时、资源竞争、回调重入。
5. 性能容量：热路径、零拷贝、大块内存搬运、重复序列化、锁内慢操作。
6. 跨平台：Windows/Linux 路径、编码、换行、大小端、编译选项。
7. 可观测性：日志、指标、trace 只有在它们是验收证据或排障契约时才列入测试。

## 用例质量标准

1. 每条用例必须能判断通过或失败。
2. 手动测试必须有操作步骤、预期结果和观测证据。
3. 自动化建议必须说明测试层级，不要只写“写测试”。
4. 不要把内部实现步骤当功能点。
5. 不要为了追数量生成低价值重复用例。
6. P0 用例数量宁可少，但必须覆盖真实风险。
