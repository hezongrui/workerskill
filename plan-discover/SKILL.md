# 需求发现与澄清 ( 头脑风暴 )

## 一句话原则
**写代码之前，必须先搞清楚要做什么。**
再小的需求（比如改个配置、加个按钮）也要先聊清楚，别让“我以为”变成返工。

## 什么时候用
- 用户说“帮我做个功能”“加个组件”“改一下这里”
- 任何需要写代码、改行为、加逻辑的需求
- **没有例外**：哪怕是改一行配置，也要走这个流程（流程可以很短，但必须有）

## 执行步骤（按顺序做）

### 1. 看上下文
先看看项目里有什么文件、最近改了什么、有没有相关文档。

### 2. 问清楚（一次只问一个问题）
用简单的话问用户：
- 这个需求的目的是什么？
- 有什么限制条件？
- 做成什么样算成功？

> 小技巧：能给出 A/B/C 选项时，尽量给选择题，不要老让用户写小作文。

### 3. 拆解大需求
如果用户说的需求很大（比如“做一个带聊天、文件、支付的系统”），**立刻叫停拆细节**，先帮用户拆成几个独立的小模块，定好先做哪一个。

### 4. 给出 2-3 个方案
每个方案说清优缺点，告诉用户你推荐哪一个，以及为什么。

### 5. 讲产品方案
分段讲，每讲完一段就问用户“这样对吗？”，确认后再讲下一段。
要覆盖：
- 用户流程（谁用、怎么用、步骤是什么）
- 功能边界（做什么、不做什么）
- 交互逻辑（界面怎么反馈、异常怎么提示）
- 成功标准（做成什么样算完成）

> 注意：这里只谈**产品层面的方案**，绝对不涉及技术实现、架构设计、文件拆分——这些留给后续的设计和规划 skill。

### 6. 写设计文档
把定下来的设计写成文档，保存到 `docs/specs/YYYY-MM-DD-<主题>-design.md`
（如果用户指定了别的路径，按用户的来）

### 7. 审设计文档
通过 `Agent` 工具派遣一个独立的 reviewer agent 审一遍设计文档，有问题就改，再审，最多审 3 轮。还不过就交给人类用户定夺。

### 8. 给用户看
让用户确认设计文档有没有问题，**用户说“没问题”之后，才能进入下一步。**

### 9. 下一步

澄清结束后，将整理好的结论（背景、角色、场景、功能边界、优先级）交给用户确认，由用户决定后续方向。

## 视觉辅助（可选）
如果聊到的东西有 UI、布局、流程图，你可以问用户：
> “后面我们可能会聊到一些界面、流程图之类的视觉内容，我可以生成 Markdown/ASCII 图或 HTML 文件来可视化展示，你想试试吗？”

- 这句话要**单独发一条消息**，别和其他问题混在一起。
- 用户同意后，每个问题自己判断：这个问题用图讲更清楚，还是文字就够了？

## 关键原则

| 原则 | 说明 |
|------|------|
| 一次只问一个 | 别一次性抛一堆问题给用户 |
| 优先给选择题 | A/B/C 比开放题好答 |
| 没必要的功能砍掉 | 别为了“专业”加一堆用不到的东西 |
| 先确认再继续 | 每讲完一段设计就问对不对 |
| 小文件优先 | 一个文件只做一件事，大了就拆 |

## 常见错误

- **“这个太简单了，不用聊”** → 再小的需求也可能有坑，聊两句不会吃亏。
- **没写文档就直接干** → 必须写设计文档并让用户确认。
- **同时抛三四个问题** → 用户容易漏答，信息不全。
