# Subagent 驱动开发

## 一句话原则
**每个任务都派一个干净的 subagent 去做，做完后先后经 spec 审查和代码质量审查，才能进入下一个任务。**

## 为什么要用
- 每个任务都有全新上下文，不会被之前的讨论污染
- 自动做 TDD、自动写测试
- 每个任务都有质量检查点，问题早发现早修
- 你在主会话里只负责协调，不用被实现细节淹没

## 什么时候用 vs 其他 skill

- **有实施计划 + 任务独立 + 在当前会话做** → 用本 skill
- **有实施计划 + 要在隔离的 session/工作区里做** → 使用隔离工作区或新 session 执行，配合本 skill 的协调流程
- **没有计划** → 先用 `plan-discover` + `plan-prd`，再进入执行

## 完整流程

### 准备阶段
1. 读一遍计划文件
2. 提取所有任务文本和上下文信息
3. 用当前环境可用的任务/进度跟踪工具（如 `TaskCreate`、`TaskUpdate` 等）列出所有任务，执行中持续更新进度

### 每个任务的循环

#### Step 1：派遣实现者 agent
- 用 `Agent` 工具派遣实现者 agent，提供完整任务文本 + 上下文
- 明确目标、限制条件和预期输出
- agent 可能会先问你问题，回答清楚再让它开始

#### Step 2：实现者完成

实现者会：
- **写代码实现任务需求**，实现过程中**直接调用 `test-driven-development` skill**；由该 skill 判断是否需要红绿循环，若需要则严格遵循（先写失败测试 → 跑测试确认失败 → 写最小实现 → 跑测试确认通过），若不需要则按常规方式写实现
- 跑测试、自我审查并保存变更

#### Step 3：Spec 合规审查
- 用 `Agent` 工具派遣 **spec-reviewer agent**
- 审查代码是否符合计划要求（功能做对了吗？边界覆盖了吗？）
- **不通过 → 实现者修 → 再审 → 直到通过**

#### Step 4：代码质量审查
- 用 `Agent` 工具派遣 **code-quality-reviewer agent**
- 审查代码写得是否干净、有没有坏味道（命名、结构、重复、可读性）
- **不通过 → 实现者修 → 再审 → 直到通过**

#### Step 5：标记任务完成
用当前环境可用的任务/进度跟踪工具更新当前任务状态为已完成。

### 全部任务完成后
1. 用 `Agent` 工具派遣一个 **final-reviewer agent**，审查整体实现
2. 通过后，做收尾（合并变更、清理隔离区等）

## 并行加速模式（Claude Code 多 Agent）

Claude Code 支持在**一条消息中同时派遣多个 `Agent`**。当计划中的任务**完全独立**（不修改同一文件、无状态依赖）时，可以分组并行执行以加速：

### 分组原则
- 按模块/文件分组，确保组之间**没有共享文件**
- 每组内部仍保持：实现 → spec 审查 → 代码质量审查的串行顺序
- 多组可以同时在各自隔离的上下文里执行

### 执行方式
1. 在一条消息中同时派遣多个实现者 agent（每组一个）
2. 等待所有实现者返回
3. 再同时为每组派遣 **spec-reviewer** 和 **code-quality-reviewer**
4. 全部组审查通过后，再进入下一批

### 什么时候不要并行
- 任务之间有依赖（A 的输出是 B 的输入）
- 可能修改同一文件
- 共享数据库 schema、配置文件或全局状态
- **你不确定两个任务是否有冲突/共享状态时 → 不要强行拆成两个 agent，优先交给同一个 agent 串行完成，或默认整体串行执行**


## 实现者返回的 4 种状态

| 状态 | 含义 | 你怎么办 |
|------|------|----------|
| DONE | 完成 | 进入 spec 审查 |
| DONE_WITH_CONCERNS | 完成但有疑虑 | 先看 concerns，严重就先处理再审查 |
| NEEDS_CONTEXT | 缺少信息 | 补信息，重新派遣 |
| BLOCKED | 卡住做不了 | 给更多信息 / 换更强模型 / 拆小任务 / 问用户 |

## 绝对不能做的事

- 跳过 spec 审查或代码质量审查
- 带着未修复的问题进入下一个任务
- 同时派多个实现者 agent **修改同一文件**（会冲突）
- 让 agent 自己读计划文件（你把任务文本直接塞给它）
- 先代码质量审查再审 spec（顺序错了）
