# Bug 整理

## 一句话

先把 bug 说清楚，再进入修复。重点是把现象、复现、影响范围、期望结果和实际结果整理清楚。

## 适用场景

1. 线上 bug、测试环境缺陷、回归问题。
2. 用户反馈“有问题”，但现象和触发条件还不清楚。
3. 准备进入修复，但还不能直接写失败测试复现问题。

## 不做什么

1. 不在这里直接做架构设计。
2. 不在这里直接展开大段技术方案。
3. 不在这里直接写修复方案、回归方案或复盘结论。
4. 不在这里替代 `code-review` 或其他后续开发动作。

## 输出路径

保存到：

`docs/bugs/[bug-name].md`

如果只是当前对话临时整理，也可以直接在对话中输出。

## 执行步骤

1. 写清 bug 现象：用户看到了什么，系统表现成什么。
2. 写清大白话描述：不用技术术语，直接说明“做了什么之后出了什么问题”。
3. 写清期望结果：正确行为应该是什么。
4. 写清实际结果：实际出现了什么现象、报错或异常行为。
5. 写清复现步骤：输入、操作、环境、前置条件。
6. 写清复现特征：必现、偶现、特定条件下出现，还是运行一段时间后出现。
7. 写清影响范围：影响谁、影响哪些环境、影响哪些模块或用户。
8. 写清证据：日志、报错、截图、返回码、时间点、相关代码位置。
9. 检查描述是否足够让后续排查和复现继续进行。

## 输出模板

```markdown
# Bug 整理 - [bug 名称]

## 1. 问题现象
- 现象：
- 出现场景：
- 首次发现时间：

## 2. 大白话描述
- 做了什么：
- 期望看到什么：
- 实际出了什么问题：

## 3. 期望结果
- 正确行为：

## 4. 实际结果
- 实际行为：
- 报错 / 返回码 / 日志：

## 5. 复现步骤
1. ...
2. ...
3. ...

## 6. 复现特征
- 必现 / 偶现：
- 触发条件：
- 是否和时间、数据量、并发、环境有关：

## 7. 影响范围
- 影响环境：
- 影响对象：
- 是否稳定复现：
- 严重程度：

```

## 输出要求

1. 先写事实，再写判断。
2. 必须补一段大白话描述，让不了解上下文的人也能快速看懂。
3. 复现步骤要可执行，不能只写“偶发失败”。
4. 要明确这是必现、偶现，还是特定条件下出现。
5. 这里只整理问题，不展开原因判断和修复方案。
