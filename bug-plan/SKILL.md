# Bug 整理

## 一句话

先把 bug 说清楚，再进入排查。重点是把现象、复现、环境、影响范围、期望结果和实际结果整理成可确认的事实。

## 适用场景

1. 线上 bug、测试环境缺陷、回归问题。
2. 用户反馈“有问题”，但现象和触发条件还不清楚。
3. 准备进入修复，但还不能直接写失败测试复现问题。

## 不做什么

1. 不在这里直接做架构设计。
2. 不在这里直接展开大段技术方案。
3. 不在这里直接写修复方案、回归方案或复盘结论。
4. 不在这里替代 `code-review` 或其他后续开发动作。
5. 不在这里记录代码定位、根因假设、修复思路——这些属于下游排查阶段。

## 输出路径

保存到：

`docs/bugs/[bug-name].md`

命名示例：

- `docs/bugs/login-token-expire.md`
- `docs/bugs/api-timeout-under-load.md`
- `docs/bugs/data-sync-missing-rows.md`

如果只是当前对话临时整理，也可以直接在对话中输出。

## 执行步骤

整理不是一次成型。信息不全时 **反复追问、确认表述、修订文档**，直到满足 Gate 或待补充项已明确。

### 每轮迭代（信息不全时重复）

1. 根据当前已知信息，输出或 **更新** 整理文档（可先对话展示，再写入 `docs/bugs/[bug-name].md`）。
2. 标注 **待补充项**，并自检能否写出 Gate 句；写不出则 **不得** 标记可交接。
3. 从 [澄清问题清单](#澄清问题清单) 中选取 **最优先的 2~3 个问题** 追问，不要一次抛出过多问题。
4. **复述当前理解**，请用户确认表述是否清晰，例如：
   - 「以上描述是否准确？有没有遗漏或说错的地方？」
   - 「非当事人能否据此理解问题？哪里还不清楚？」
5. 根据用户反馈 **修订文档**：补事实、去歧义、合并重复描述，更新待补充项。
6. 重复以上步骤，直到 Gate 可写出且用户确认清晰，或暂时无法获取的信息已明确列入待补充项。

### 单轮填写要点

1. 向用户/相关方收集必要信息，确保不是凭片段猜测。
2. 写清 bug 现象：用户看到了什么，系统表现成什么。
3. 写清大白话描述：不用技术术语，直接说明“做了什么之后出了什么问题”。
4. 写清期望结果：正确行为应该是什么。
5. 写清实际结果：实际出现了什么现象、报错或异常行为。
6. 写清复现步骤：输入、操作、环境、前置条件。
7. 写清复现特征：必现、偶现、特定条件下出现，还是运行一段时间后出现。
8. 写清影响范围：影响谁、影响哪些环境、影响哪些模块或用户、严重程度。
9. 写清环境信息：版本、分支、部署环境、相关配置、首次发现时间、近期变更。
10. 记录已掌握的证据：日志、报错、截图、返回码、时间点。只记录原始信息，不做解读或推断；尚未掌握的证据标注为"待补充"。
11. 写出 Gate 句并自检：「当我 [动作] 时，期望 [X]，实际 [Y]。」写不出则回到追问，不得进入 `bug-investigation`。
12. 检查整理是否完整：一个不了解上下文的人能否根据本文确认并尝试复现该问题。

### 何时可结束整理

- **可交接**：Gate 已写出，关键事实无歧义，用户确认描述清晰。
- **暂停交接**：仍有待补充项且暂时无法获取——文档标注待补充，**不**进入 `bug-investigation`。
- **禁止勾选**「可进入 bug-investigation」：信息明显不全、Gate 写不出、用户未确认清晰时。

## 澄清问题清单

追问时按优先顺序选取：**环境 → 复现 → 影响 → 证据 → 变更**。

bug 整理阶段常需要追问用户或相关方：

- 这个问题首次出现是什么时候？上线后一直有，还是某次变更后才出现？
- 哪个环境发现的？生产、测试、本地？
- 影响的是全部用户，还是特定用户/角色/区域？
- 是必现、偶现，还是只在特定条件下出现？
- 是否有稳定复现步骤？最小复现路径是什么？
- 有没有相关日志、报错截图、返回码或监控截图？
- 最近有没有发布、配置变更、数据迁移、依赖升级？
- 业务上的影响是什么？有没有临时规避方案？

## 输出要求

1. 先写事实，再写判断。
2. 必须补一段大白话描述，让不了解上下文的人也能快速看懂。
3. 复现步骤要可执行，不能只写“偶发失败”。
4. 要明确这是必现、偶现，还是特定条件下出现。
5. 证据记录已掌握的原始信息（日志、报错、截图、返回码、时间点），缺失项明确标注"待补充"；不解读、不定位代码、不推断根因。
6. 若关键信息暂时无法获取，先输出当前整理结果，并列出待补充项；**继续追问**，不要求一次整理完整。
   - 必须继续追问的情况：Gate 句写不出、用户未确认描述清晰、关键事实存在歧义、待补充项可能影响根因定位。
7. 信息不全时 **反复追问**：每轮最多 2~3 个问题，收到回答后 **更新文档**，并向用户确认表述是否准确、清晰。
8. 整理完成前必须写出 Gate 句：「当我 [动作] 时，期望 [X]，实际 [Y]。」写不出则继续追问，不得进入 `bug-investigation`。
9. 多轮整理时在「整理修订」中简要记录每轮更新了什么、用户是否确认，便于后续接续。
10. 仅当文档状态为「可交接」且 Gate、用户确认均已满足时，才交接给 `bug-investigation`。
11. 这里只整理问题，不展开原因判断和修复方案。

## 输出检查清单

- [ ] 大白话描述完整，非技术人员也能看懂问题
- [ ] 期望结果和实际结果已分开写清
- [ ] 复现步骤可执行，前置条件明确
- [ ] 已明确必现 / 偶现 / 特定条件
- [ ] 影响范围包含环境、对象、规模、严重程度
- [ ] 环境信息包含版本、部署环境、近期变更
- [ ] 证据只记录已掌握的原始信息，缺失项已标注"待补充"
- [ ] Gate 句已写出：「当我 [动作] 时，期望 [X]，实际 [Y]。」
- [ ] 用户已确认描述准确、清晰（或待补充项已明确标注）
- [ ] 信息不全时已追问并更新文档，未假装一次整理完成
- [ ] 状态为「可交接」时才勾选进入 `bug-investigation`
- [ ] 全文不涉及修复方案、回归方案、复盘结论
