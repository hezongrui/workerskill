# onboard-architecture

## 说明

对应《老项目快速上手 Playbook》阶段 1：全局架构扫描。

本 skill 产出**项目全景认知文档**，目标是让新人在 5 分钟内看懂：
- 这个项目解决什么业务问题？
- 有几个模块？各是干嘛的？
- 数据怎么流进来的？又怎么流出去的？
- 用了什么技术？

**严禁涉及具体代码细节**（函数实现、代码片段、方法名、参数列表、算法逻辑）。
**核心类名允许保留**——只列出核心类的名称和一句话职责，作为从目录到代码的导航锚点。

## 执行流程（共 3 轮）

### Step 1: 并行扫描各模块（第 1 轮）

主 Agent 快速浏览根目录和一级子目录，按功能域将代码库拆分为 2~4 个扫描单元。

对每个单元，**并行启动独立子 Agent**。

**重要：Agent 失败处理机制**
- 并行启动后，**逐个检查每个 Agent 的返回结果**，不要一次性等全部返回后才处理。
- 若某个 Agent 返回失败/被拒绝/超时无结果，**主窗口立即重新启动该 Agent**（复用相同 prompt，无需等待其他 Agent）。
- 若同一 Agent 连续失败 2 次，**主窗口降级为直接读取该单元源码**，自己完成分析并写入文档，不再依赖子 Agent。
- 始终保持主窗口上下文：每处理完一个单元（无论通过 Agent 还是主窗口自己），立即更新进度并继续下一个，禁止空等待。

Prompt 模板：
```
你是项目架构分析专家。请仅扫描以下单元的宏观信息。

**严禁输出具体代码细节**（函数实现、代码片段、方法名、参数列表、算法逻辑）。
**允许列出核心类名 + 一句话职责**。

单元名：[单元名]
单元路径：[单元路径]

严格按以下 5 项输出：
1. 一句话职责：这个模块是做什么的（从业务角度说人话）
2. 目录结构：该单元内的目录层级，每个目录一句话职责
3. 技术栈：编程语言、框架、关键依赖库
4. 核心类：列出承担核心职责的类/结构体，每个类名 + 一句话职责
5. 数据流：这个模块的数据从哪来、到哪去、经过什么关键组件。格式示例：外部设备(TCP) → [LogRecverTcpComm] → [磁盘文件] → [FetchLogHttpSrv] → [外部HTTP客户端]

约束：
- 只分析 [单元路径] 内结构
- 不猜测外部目录细节
- 不确定项标注"待确认"
```

### Step 2: 汇总审查（第 2 轮）

启动汇总审查 Agent，整合各单元结果：
- 模块间的依赖关系和接口边界
- 全链路数据流一致性
- 技术栈和构建系统层级一致性

审查要点：
- 事实准确性：路径、版本、文件位置是否正确？
- 逻辑一致性：模块划分是否反映真实依赖？
- 纯净度检查：是否意外出现了代码细节？
- 可执行性：新人能否拿着文档快速找到目录？

输出：整合后的架构草案 + 事实性错误清单 + 结论【通过 / 补充修正】

### Step 3: 输出最终文档（第 3 轮）

综合扫描结果，输出最终版 `docs/onboarding/architecture.md`。

文档结构（严格按此顺序，精简输出）：

1. **业务场景（置顶，必须先说人话）**：
   - 这个项目解决什么业务问题？面向谁？典型使用场景是什么？
   - 用 1~2 个具体例子说明（如"某智能家居设备出现故障，运维人员通过本系统查询该设备上传的日志"）
   - 30秒速览：3句话以内，说明"这是什么"，包含核心数字（几个模块、几个入口、用什么核心协议）

2. **功能模块一览**：
   - 表格形式：模块名 | 一句话职责 | 新人优先级【必须先看 / 了解即可 / 排查问题时再看】
   - 只列核心模块，周边系统收进"周边依赖"小节

3. **数据流向图**：
   - 一张图说明：数据从哪来、经过哪些模块、到哪去
   - 标注外部系统边界（设备 / 浏览器 / 下游服务）
   - 标注核心数据格式或协议

4. **技术栈**：
   - 表格形式：层级 | 技术 | 用途
   - 只列关键项，不展开版本和配置细节

5. **目录结构**：
   - 只列出核心目录，每个目录一句话职责
   - 标注【新手先看】/【了解即可】/【改 bug 时再看】

6. **架构分层**：
   - 1~2 句话说明分层逻辑
   - 简单的层间关系图（如 接入层 → 业务层 → 存储层）

7. **核心类清单**：
   - 表格形式：类名 | 一句话职责 | 所在文件

8. **构建与运行**：
   - 怎么编译（一句话）
   - 运行入口在哪（文件路径）
   - 产出了什么可执行文件

9. **配置与测试**：
   - 关键配置文件位置和作用
   - 测试在哪、用什么框架

通用要求：
- 先说人话，再谈技术。业务场景必须在技术细节之前。
- 只保留确定无误的宏观事实，删除推测性描述。
- 删除所有具体代码细节，保留核心类名 + 一句话职责。
- 新人看完应该能回答："这个项目是做什么的？有哪几个大模块？数据怎么走的？"

- `docs/onboarding/architecture.md`
