# onboard-architecture

## 一句话

先看全局，再看细节。这个 skill 只负责产出项目全景图，帮助新人快速回答：项目是做什么的、主要模块有哪些、主链路怎么走。

## 适用场景

1. 刚接手老项目，想先看懂整体结构。
2. 项目代码量较大，不适合一上来就扎进单个文件。
3. 后续准备继续做模块拆解或重点模块深挖。

## 不做什么

1. 不展开具体代码实现细节。
2. 不深挖单个模块内部逻辑。
3. 不直接替代 `onboard-modules` 或 `onboard-focus`。

## 执行方式

默认使用单 agent 顺序分析。

如果项目特别大、一级目录很多、模块边界比较清楚，可以按一级目录或功能域拆成 2 到 4 组做并行粗扫；但并行只用于加速收集宏观信息，最终结论仍由主 agent 统一汇总。

## 看什么

1. 项目解决什么业务问题。
2. 核心目录和核心模块分别负责什么。
3. 主链路或主数据流怎么走。
4. 使用了哪些关键技术、构建方式和运行入口。
5. 新人应该先从哪里看起。

## 输出路径

保存到：

`docs/onboarding/architecture.md`

如果只是当前对话临时分析，也可以直接在对话中输出。

## 输出要求

1. 先说业务，再说技术。
2. 只保留确定无误的宏观事实，不写猜测。
3. 可以保留核心类名，但只写类名和一句话职责。
4. 看完文档后，新人应该能回答：
   - 这个项目是做什么的？
   - 有哪几个大模块？
   - 主链路怎么走？
   - 先从哪里看起？
