# 错误复盘沉淀

## 一句话

不是为了写总结，而是为了把“这次错过一次”的代价，换成以后少错一次的规则。

## 输出路径

单次复盘：
`docs/retrospectives/YYYY-MM-DD-<topic>.md`

长期模式库：
`docs/retrospectives/patterns.md`

## 适用边界

适合：

1. 线上 bug 修复完成后。
2. code review 发现严重缺陷后。
3. CI / 构建 / 测试失败定位完成后。
4. 遗留问题、回归问题、性能回退处理完成后。
5. AI 辅助开发后结果需大改、返工、弃稿，或审查发现 AI 引入的系统性问题时。

不适合：

1. 问题还没定位清楚时先写复盘。
2. 纯吐槽式总结，没有可执行改进行动。
3. 只有「AI 不好用」的笼统感受，说不清期望 vs 实际差异、具体 diff 或决策点。

## 核心原则

1. 复盘问题，不复盘人。
2. 写事实链，不写情绪链。
3. 每条复盘都要落到一个或多个可执行动作。
4. 动作优先反哺现有 skill：测试、审查、交付、架构、日志、CI、任务切分与 AI 使用边界。

## 复盘顺序

1. 现象：发生了什么。
2. 影响：影响了谁，范围多大。
3. 根因：代码、设计、流程、环境哪个环节出了问题。
4. 漏检原因：为什么 TDD、测试计划、review、CI 没提前拦住。
5. 改进行动：补测试、补 review 检查项、补日志、补告警、补文档、补 skill。
6. 模式归档：把这类问题抽象成可复用模式。

## 常见错误模式

1. 生命周期和资源释放遗漏。
2. 边界条件和错误路径未测。
3. 并发和时序假设错误。
4. 外部契约变化未覆盖。
5. PR 收口不足，测试证据不全。
6. 环境或配置差异未显式声明。
7. 性能优化改动引入语义回归。

## AI 辅助开发常见失误模式

1. 任务一次给太大，缺少 `plan-dev` / 阶段边界，输出扩散到无关文件。
2. 该用 skill 时没用（如缺 `test-plan-design`、缺架构约束），或上下文里缺项目规范。
3. 幻觉 API、臆测未读代码、编造不存在的模块/配置/寄存器行为。
4. 过度设计：多余抽象层、线程池、通用框架，违反最小增量。
5. 只保证 host 编译通过，忽略 target、交叉编译、上机/联调项。
6. 嵌入式/中间件约束遗漏：ISR 阻塞、锁粒度、资源释放、时延热路径。
7. 审查门禁不足：只看「能编译」，未审并发/契约/测试证据就差点合并。
8. 把 AI 输出当最终稿，未走 `code-review` / `pr-delivery` 收口。

### ai-delivery 专用补充（问题类型为 ai-delivery 时必填）

第 4、5 节除通用项外，还必须覆盖下表：


| 维度          | 要写什么                                           |
| ----------- | ---------------------------------------------- |
| 期望 vs 实际    | 本来要什么；AI 给了什么；差在哪（方向错 / 漏项 / 过度设计 / 幻觉 / 不可上机） |
| 任务切分        | 是否一次让 AI 做太多；是否缺 PRD 阶段或 `plan-dev` 文件边界       |
| Skill / 上下文 | 该用哪个 skill 没用；是否缺架构/测试计划/项目规范/嵌入式约束            |
| 验证          | 是否未要求 host 测或上机项就接受；是否跳过 `pr-delivery` 自审      |
| 审查          | 你是否只看了编译通过；哪些 P0 项（并发/资源/契约）应拒收                |
| 工具边界        | 是否让 AI 做了不适合的事（猜硬件、猜未读代码、跨模块大 refactor）        |


改进行动至少一条必须落到：**具体 skill 修改**、**具体 review 检查项**、**流程调整（如先 test-plan 再 TDD）** 或 **明确此类任务不用 AI 自动生成** 四者之一。

## 反哺规则

1. 能补自动化测试的，优先补测试。
2. 不能自动化测试但能审查发现的，补进 `code-review` 检查项。
3. 能在提交前发现的，补进 `pr-delivery`。
4. 能在 CI 发现的，补进 `ci-failure-triage` 或流水线检查。
5. 只是个体经验、无法复用的，不要硬写成规则。
6. AI 返工类：能写成审查清单的，优先补进 `code-review`；能写成提交前门禁的，补进 `pr-delivery`；能写成任务边界的，补进 `plan-dev` / `plan-prd`；能写成「此类不用 AI」的，写入项目规则或对应 skill。
7. AI 失误若仅因单次 prompt 口误、不可复现，写进单次复盘即可，不进入 `patterns.md`。

## patterns.md 维护方式

长期模式库按下面格式累积：

```markdown
## [错误模式名]
- 典型信号：
- 常见根因：
- 优先拦截位置：TDD / Review / CI / Delivery / Runtime / Skill选择 / 人工门禁
- 建议动作：
```

AI 类模式可在模式名前缀加 `AI-`，例如 `AI-过度抽象中间件`、`AI-忽略target验证`。

## 完成标准

- [ ] 已写清现象、影响、根因
- [ ] 已说明为什么没提前发现
- [ ] 已提出可执行改进行动
- [ ] 已明确反哺到哪个现有 skill 或流程
- [ ] 已判断是否需要加入长期模式库
- [ ] （ai-delivery）已写清期望 vs 实际，且至少一条动作落到 skill / review / 流程 / AI 边界
