# 测试驱动开发

## 一句话

先用一个失败的测试描述期望行为，再写刚好让它通过的实现，然后在测试保护下整理设计。

TDD 不是宗教。它是一种把需求、接口和回归风险压缩进短反馈循环的开发习惯。

## 使用入口

进入本 skill 后先做 4 件事：

1. 判断任务类型：bug、重构、新行为、边界契约、补覆盖率或探索工作。
2. 判断技术栈：C++ 读取 `references/cpp.md`，Python 读取 `references/python.md`。
3. 识别项目现有测试框架和命令：优先沿用项目已有结构，不另起一套测试体系。
4. 把需求写成行为卡片，再开始红绿循环。

行为卡片模板：
```text
场景：
输入 / 初始状态：
执行的公开接口：
期望的可观察结果：
测试层级：
断言对象：
不测试的内容：
```

如果行为卡片里的“期望结果”写不成可断言结果，先澄清需求，不要强行写测试。

## TDD 优先级

按下面顺序判断。`P0` 必须严格测试先行，`P1` 强烈测试先行，`P2` 选择性测试先行，`P3` 不为测试而测试。

1. **P0 线上 bug / 已知缺陷修复**：先写失败测试复现问题，再修复。这个测试必须留下来做回归保护。
2. **P0 重构和遗留代码改造**：先补 characterization tests 锁住当前行为，再改结构。重构期间不新增行为。
3. **P1 核心业务行为**：算法、状态流、解析器、校验器、策略选择、错误处理，优先用单元测试驱动。
4. **P1 外部边界契约**：文件、网络、消息、进程、设备、第三方 SDK 等，优先测自己的封装和协议，不测第三方库本身。
5. **P1 异常和边界条件**：空值、非法输入、重复调用、超时、取消、幂等、资源释放，要在 happy path 之后尽快补。
6. **P2 集成路径**：关键模块组合、真实 fixture、小型端到端路径，用较少但可信的测试覆盖。
7. **P3 覆盖率补洞**：只补有业务风险的洞，不为追数字写低价值测试。
8. **P3 探索类工作**：spike、性能试验、第三方 API 摸底可以先实验，但实验代码不要直接合入正式路径。

可以不严格 TDD 的情况：
1. 纯配置、文档、简单脚手架。
2. 自动生成代码。
3. 视觉/交互微调，除非项目已有快照或端到端测试规范。
4. 需求还没有形成可验证行为时，先澄清行为，再写测试。

正式代码进入主线前，应回到测试保护下实现关键行为。

## 核心循环

1. 写一句行为说明：在什么条件下，调用什么公开接口，应该得到什么结果。
2. 选择稳定边界：优先测公开函数、类接口、模块 API、CLI/API 行为，不测私有细节。
3. 红：写一个最小失败测试，只覆盖一个行为。
4. 验红：确认失败原因正确，不是语法错误、环境错误或测错对象。
5. 绿：写最小实现让这个测试通过，不顺手加“以后可能用到”的功能。
6. 验绿：跑当前测试，再跑相关测试，必要时跑全量测试。
7. 重构：只整理命名、结构、重复代码，不改变行为。
8. 下一个行为：重复上面的循环。

好节奏是“小步快跑”。一个测试通常应该在几分钟内完成红到绿。

## 可量化断言优先级

TDD 必须落到可观察结果。按下面顺序找断言对象：

1. **返回值**：输入 A，返回 B。
2. **错误结果**：非法输入返回错误码、`None`、`expected`、异常或明确失败状态。
3. **状态变化**：对象、状态机、数据库记录、缓存内容从 A 变成 B。
4. **持久化输出**：生成文件、配置、记录、消息 payload，断言稳定字段或内容。
5. **边界交互**：对外部系统发起一次明确调用，例如发送命令、写入队列、启动进程。
6. **事件和指标**：业务事件、告警、metric 名称和值是系统契约时可以断言。
7. **日志**：只有日志本身是契约时才断言。
8. **性能和时序**：放在独立性能/集成测试里，不用脆弱的单元测试硬卡真实时间。

不要用“代码看起来更清晰”“应该更健壮”当测试目标。先把它翻译成上面某一种可断言结果。

## 日志断言优先级

日志是诊断工具，不是默认测试目标。按下面顺序判断：

1. **P0 审计 / 安全 / 合规日志**：必须测。断言事件类型、操作者、对象、结果、错误码等稳定字段。
2. **P0 运维告警和结构化事件**：必须测。断言 level、event、error_code、关键上下文和次数。
3. **P1 错误处理契约日志**：建议测。比如异常被吞掉但必须记录 `request_id`、`device_id`、`operation`、`reason`。
4. **P2 关键业务状态日志**：可测。只有当团队把它当作业务事件或排障契约时才测。
5. **P3 普通 info/debug 日志**：通常不测。它们是诊断细节，变化频繁。
6. **P3 完整自然语言文案**：不要测。只测稳定字段，不测整句描述。
7. **P3 时间戳、行号、线程号、文件名**：不要测。这些是运行环境细节。
8. **P3 为了证明内部分支执行过而写的日志**：不要测。应改测返回值、状态、边界交互或事件。

如果必须测日志，断言顺序是：
1. 日志级别。
2. 事件名或错误码。
3. 必要上下文字段。
4. 次数。

## 测试分层

日常开发优先顺序：
- 单元测试：纯逻辑、值对象、状态转移、错误分支，数量最多，运行最快。
- 组件测试：模块边界上的组合行为，使用 fake/stub 隔离慢依赖。
- 集成测试：验证文件系统、数据库、网络、第三方 SDK、进程等真实边界。
- 端到端测试：只覆盖最关键主路径，数量少，放在 CI 或发布前关口。

不要用覆盖率替代判断。覆盖率是提示，不是目标。更重要的是关键行为、异常路径、边界条件和回归用例是否被保护。

## Test Double 策略

默认测真实行为，谨慎替换依赖。

优先级：
1. 真实对象：纯内存、快、稳定时优先。
2. Fake：轻量可运行替身，例如内存仓库、临时目录、假时钟。
3. Stub：固定返回值，用来驱动特定路径。
4. Mock：只在需要验证跨边界交互契约时使用，例如“必须发出一次取消请求”。

避免：
- mock 被测类本身
- 为了测试私有步骤而 mock 内部函数
- 断言实现步骤多于断言可观察结果
- mock setup 比测试意图还长
- 只填当前断言用到的 mock 字段，导致替身和真实对象结构不一致

可以验证交互的情况：
- 调用外部系统、消息总线、文件写入、进程启动、计费、告警、硬件/设备命令等边界行为
- 重试、超时、幂等、取消、资源释放等协议性行为

## 日常工作流

新功能：
1. 先列 2 到 5 个行为点，按风险排序。
2. 从最小可验证行为开始写第一个测试。
3. 每个测试通过后再决定下一个行为，不一次性铺满测试。

修 bug：
1. 写一个失败测试复现 bug。
2. 确认失败信息与线上/用户现象一致。
3. 最小修复。
4. 保留该测试作为回归保护。

重构：
1. 先确认已有测试足够覆盖目标行为。
2. 不足时先补 characterization tests，记录“当前行为是什么”。
3. 重构时不新增行为。
4. 每个小步后跑相关测试。

遗留代码：
1. 找到最稳定的外部入口。
2. 先补粗粒度测试锁住现有行为。
3. 再逐步抽小模块，给新模块补更快的单元测试。

## 好测试标准

一个好测试应该：
- 名字说明行为，而不是说明实现步骤
- 独立、可重复、顺序无关
- 快，失败信息清楚
- setup 少，意图明显
- 不依赖真实时间、随机数、网络、共享全局状态
- 对时间使用假时钟，对文件使用临时目录，对随机使用固定种子
- 只在日志本身是契约时断言日志；日志是诊断工具，不是进入 TDD 的前置条件

推荐结构：
```text
Arrange: 准备输入和依赖
Act: 调用被测公开接口
Assert: 断言可观察结果
```

## 常见反模式

- 先写实现，再补测试：测试容易变成给实现找理由。
- 批量写很多测试，再批量实现：反馈太慢，容易测错范围。
- 只测 happy path：缺少边界、异常、空值、重复调用、并发/超时路径。
- 测私有方法：重构时测试跟着碎。
- 过度 mock：测到的是替身剧本，不是真实行为。
- 追求 100% 覆盖率：容易制造低价值测试。
- flaky test：偶发失败的测试会消耗团队信任，必须优先修。

## 团队落地规则

1. **PR 快速门禁**：每次提交至少跑相关单元测试和静态检查，失败不合入。
2. **CI 分层**：PR 跑快测试；主干跑全量单元和组件测试；夜间或发布前跑慢集成、端到端和性能回归。
3. **flaky test 处理**：偶发失败按缺陷处理。要么修稳定，要么隔离到慢套件并登记原因，不能长期忽略。
4. **覆盖率原则**：覆盖率只做趋势和风险提示。核心模块可以设基线，但不追求无意义的 100%。
5. **代码评审标准**：评审重点看测试是否描述真实行为、是否稳定、是否过度 mock、是否污染生产接口。
6. **测试债务**：允许短期欠测试，但必须写清楚风险、补测试位置和触发条件，不能让债务隐形。

## 技术栈路由

本 skill 只维护日常使用的 C++ 和 Python：

| 技术栈 | 判断信号 | 读取指南 |
| --- | --- | --- |
| C++ | `CMakeLists.txt`、`.cpp`、`.hpp`、`.h` | `references/cpp.md` |
| Python | `pyproject.toml`、`pytest.ini`、`.py` | `references/python.md` |

读取语言指南后，仍以项目现有测试框架、命令、目录约定为准。指南只提供默认路径和评审标准。

其他语言按项目既有测试规范执行，不在本 skill 中维护。

## 完成定义

提交前确认：
- [ ] 每个行为变化至少有一个能失败再变绿的测试
- [ ] bug 修复保留了回归测试
- [ ] 测试断言的是公开行为或明确的边界契约
- [ ] 每个测试都有可量化断言，不靠主观描述
- [ ] 日志只在它是契约时被断言
- [ ] 没有为测试污染生产接口
- [ ] mock/fake 的层级合理，替身结构可信
- [ ] 当前相关测试通过
- [ ] 需要时已跑全量测试或 CI 等价命令
- [ ] 重构没有混入新行为

有一项不满足，就回到对应步骤补齐。
