# 开发规划协议

> **最小侵入原则**：在现有代码上加功能时，若使用者未明确要求修改现有类/界面，**优先【新增】独立类或独立界面**，而非改动现有代码。仅当新增方案会导致严重代码重复或架构冲突时，才考虑【修改】现有类。

## 三阶段分层（完整功能推荐）

> **简化模式**：对于小需求、局部改动或简单脚本，可直接使用下方【新增 / 修改 / 复用】标记来规划文件顺序，**不必强行拆成三层**。三阶段分层只建议在功能较完整、模块边界清晰时使用。

### 阶段1: 独立层（零依赖）

- **内容**：工具函数、领域模型、纯函数、配置常量
- **标准**：可独立编译/测试，无外部依赖

### 阶段2: 注入层（依赖注入）

- **内容**：业务逻辑、数据访问、API封装
- **标准**：依赖通过参数/构造注入，可mock替换

### 阶段3: 组合层（装配协调）

- **内容**：应用入口、路由、初始化逻辑
- **标准**：完成依赖装配，功能完整运行

**依赖方向**：阶段3 → 阶段2 → 阶段 1（禁止反向、禁止循环）

## 执行流程

1. **向用户收集信息** → 索要 PRD、架构文档或口述需求，理解模块划分和依赖关系
2. **分析现有代码**（如有）→ 识别可【复用】/需【修改】的部分
   - **最小侵入原则**：在现有代码上加功能时，若使用者未明确要求修改现有类/界面，**优先【新增】独立类或独立界面**，而非改动现有代码
   - 仅当新增方案会导致严重代码重复或架构冲突时，才考虑【修改】现有类
3. **确认日志方案** → 检查项目是否已有日志模块
   - 有：使用该日志模块，按现有规范输出
   - 无：询问使用者是否需要日志模块；若明确不需要，使用语言原生日志（如 Qt `qDebug()`、Python `logging`、Go `log`）
4. **按三阶段分层（可选）** → 小需求可直接按【新增 / 修改 / 复用】标记排序，不必强行拆三层；完整功能再分配到独立 / 注入 / 组合层
5. **确定实现顺序** → 同阶段内按依赖关系排序
6. **输出规划文档** → `docs/plans/dev-[project-name].md`

---

## 日志规范

- 每层开发必须输出日志，方便后续排查
- 优先使用项目现有日志模块；无则按语言原生方式输出
- 日志级别：DEBUG（详细流程）/ INFO（阶段完成）/ WARN（异常但可继续）/ ERROR（阻断错误）

## 输出规范

**路径**: `docs/plans/dev-[project-name].md`

**检查项**:
- [ ] 三阶段分层清晰（独立→注入→组合），或已明确使用简化模式仅保留【新增/修改/复用】标记
- [ ] 【新增/修改/复用】标注完整
- [ ] 依赖方向正确（无反向/循环）
- [ ] 日志方案已确认（现有模块 / 语言原生 / 明确不用）
